# Engineering Practices Skills Manifest

Use these skills when the task is primarily about software design quality, implementation discipline, testing practice, review quality or domain modelling.

| Skill | Use when |
|---|---|
| `kiss-principle` | Simplifying code, workflows or architectures that have grown beyond the requirement. |
| `solid-principles` | Clarifying responsibilities, extensibility, substitution, or dependency boundaries. |
| `dry-principle` | Removing genuinely repeated knowledge, rules, schemas or validation. |
| `tdd-practice` | Driving behaviour changes through failing tests and the smallest passing implementation. |
| `bdd-practice` | Clarifying business-visible behaviour, scenarios and acceptance criteria. |
| `ddd-practice` | Protecting domain language, invariants and bounded contexts in complex domains. |
| `code-review-and-quality` | Reviewing diffs for correctness, tests, security and maintainability before merge. |
| `git-workflow-and-versioning` | Managing branch hygiene, commit scope, release flow and versioning discipline. |

Required checks: preserve public behaviour unless intentionally changed, keep diffs small, update tests when behaviour changes, and report evidence rather than intent.
