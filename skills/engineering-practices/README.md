# Engineering Practices

Use this category when the task is mainly about code quality, design discipline, testing, refactoring or domain modelling.

## Start here when

- the hard part is implementation quality rather than workflow orchestration;
- the change needs better tests, clearer boundaries or simpler code;
- review readiness, refactoring or domain language is central.

## Skills in this category

- `kiss-principle/`
- `solid-principles/`
- `dry-principle/`
- `tdd-practice/`
- `bdd-practice/`
- `ddd-practice/`
- `code-review-and-quality/`
- `git-workflow-and-versioning/`

## Choose the smallest fit

- Start with `tdd-practice` for behaviour changes and fixes.
- Use `kiss-principle` before adding abstraction.
- Use `ddd-practice` when domain boundaries and invariants are the core issue.
- Use `code-review-and-quality` when judging whether a change is merge-ready.
